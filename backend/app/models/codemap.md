# backend/app/models/

## Responsibility

- Owns the SQLAlchemy declarative definitions for every business entity that touches the order lifecycle, delivery tooling, notifications, and admin configuration. The `Order` table is the central pivot for pick/QA/delivery/shipping work, while the surrounding tables (audit logs, status history, print jobs, delivery runs, vehicle checkouts, teams config/notifications, webhooks, users/sessions, and system settings) capture complementary state needed by services and APIs.
- Ensures the ORM layer exposes enough metadata (enumerations, timestamps, computed columns, indexed foreign keys) to let the service layer enforce SLA-driven state machines without leaking implementation details upward.

## Design

- All models inherit from `app.database.Base` and use explicit UUID primary keys, `datetime.utcnow()` defaults, and deliberate indexes (e.g., `Order` indexes on `status`, `updated_at`, `delivery_run_id`; `OrderStatusHistory` and `AuditLog` indexes on timestamps; `PrintJob` indexes on `status/created_at`). Every lookup that needs case-insensitive matching is supported (`Order.inflow_order_id_lower` computed column) so the services can filter efficiently.
- `Order` (and its `OrderStatus`/`ShippingWorkflowStatus` enums) defines the canonical fields for tagging, picklists, QA, shipping workflow metadata, delivery run assignment, and remainder handling. Relationships tie it to `AuditLog` (legacy timeline), `OrderStatusHistory` (normalized state transitions), `PrintJob` (picklist/label printing), and `DeliveryRun` (active runner assignments). The audit/history tables keep `changed_by`/`actor_user_id` references, metadata JSON blobs, and UTC timestamps to drive UI timelines and reporting.
- Additional models capture cross-cutting concerns: `SystemAuditLog`/`SystemAuditLogArchive` record every service action via `AuditService`; `PrintJob` tracks asynchronous picklist printing status (pending/claimed/completed/failed) and emits Socket.IO events when jobs change; `DeliveryRun` holds route metadata and the `orders` relationship; `VehicleCheckout` snapshots who has the keys and enforces mutexes via service-level locking; `InflowWebhook` stores registration secrets plus failure counters so the webhook handler can surface stale/failed subscriptions; `TeamsConfig`/`TeamsNotification` back the legacy Power Automate/graph-based recipient notifications; `User`/`Session` keep SAML-authenticated identities and their active sessions; `SystemSetting` exposes searchable toggles (email, Teams, auto-print) consumed by service helpers.

## Flow

- `InflowService` + webhook handlers feed orders into `OrderService` using `create_order_from_inflow`; the service extracts shipping/delivery metadata, calculates building codes, and either updates an existing `Order` row or creates a new one. Every state change goes through `OrderService.transition_status`, which validates allowed moves (see `OrderStatus` transitions), checks pre-delivery prerequisites (tagging/picklist/QA), and records modeling history in `OrderStatusHistory` plus human-facing entries in `AuditLog`. `AuditService` also writes to `SystemAuditLog` so downstream observers can track system-wide events.
- Public operations (tagging, picklist generation, QA submission, QA-triggered delivery/shipping transitions, signature capture) mutate `Order` fields, push `PrintJob` rows when needed, and enqueue Socket.IO/Teams notifications from the route layer (`orders` blueprint). `ShippingWorkflowStatus` fields are guarded by `order_service.transition_shipping_workflow`, and shipping metadata (carrier, tracking) is stored alongside the same `Order` row for UI consumption.
- `DeliveryRunService` manipulates `DeliveryRun` rows, takes over the `orders` relationship, and uses `VehicleEnum`/`DeliveryRunStatus`. `VehicleCheckoutService` serializes checks in/out of vehicles and guards against overlapping runs by querying both `VehicleCheckout` and `DeliveryRun`. `TeamsRecipientService` and `app.utils.idempotency.check_recent_notification` occasionally read/write `TeamsNotification`/`TeamsConfig` to prevent duplicates and drive notification queues.

## Integration

- `backend/app/api/routes/orders.py` is the biggest consumer of these models: it queries `Order`, related `PrintJob` entries, and `AuditLog`/`OrderStatusHistory`, streams status updates over Socket.IO, and delegates creation of picklists/QAs/status transitions to `OrderService` (which itself uses `AuditService`, `PrintJobService`, `SystemSettingService`, and `OrderStatusHistory`).
- `backend/app/api/routes/delivery_runs.py` + `DeliveryRunService` operate on `DeliveryRun`, touching the `orders` relationship, `VehicleEnum`, and emitting vehicle-status broadcasts; the service also writes to `SystemAuditLog` for administrative history.
- `backend/app/api/routes/vehicle_checkouts.py` + `VehicleCheckoutService` use `VehicleCheckout` for check-in/out history and call `broadcast_vehicle_status_update_sync` so the UI reflects real-time availability.
- `backend/app/api/routes/inflow.py` relies on `InflowWebhook` to validate signatures/failures and uses `OrderService.create_order_from_inflow` to seed `Order` rows; sync endpoints also query `Order` counts for observability.
- `backend/app/api/routes/auth.py` uses `User`/`Session` via `saml_auth_service` to create/update tamu identities, issue session cookies, list/revoke sessions, and power middleware helpers that populate `g.user_data`.
- `app.services/*.py` tie everything together: `OrderService`, `PrintJobService`, `DeliveryRunService`, `VehicleCheckoutService`, `AuditService`, `SamlAuthService`, and `SystemSettingService` each work directly with these models to enforce business invariants, queue background tasks, emit audit records, and enforce admin toggles. `teams_recipient_service` and `app.utils.idempotency` consult `TeamsConfig`/`TeamsNotification`. `backend/scripts/database_manager.py` and `alembic/env.py` import the models to populate stats/migrations.
